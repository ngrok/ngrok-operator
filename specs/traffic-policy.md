# Traffic Policy

> The ngrok traffic policy DSL used to define rules for processing traffic at endpoints.

<!-- Last updated: 2026-04-08 -->

## Overview

The `internal/trafficpolicy` package provides Go types and builders for the ngrok traffic policy configuration language. Traffic policies define rules for each phase of a connection's lifecycle and are applied to endpoints via the ngrok API.

Traffic policies can be specified in three ways:
1. **Inline on CRDs**: `AgentEndpoint.spec.trafficPolicy.inline` or `CloudEndpoint.spec.trafficPolicy`.
2. **Shared via NgrokTrafficPolicy CRD**: Referenced by name from endpoint CRDs.
3. **Generated by the Manager Driver**: Constructed programmatically during IR-to-endpoint translation (e.g., routing rules from Ingress/Gateway API resources).

## Structure

```
TrafficPolicy
├── on_http_request: Rule[]
├── on_http_response: Rule[]
└── on_tcp_connect: Rule[]

Rule
├── name: string (optional)
├── expressions: string[] (CEL expressions, all must match)
└── actions: Action[]

Action
├── type: ActionType (e.g., "add-headers", "forward-internal")
└── config: object (type-specific configuration)
```

## Phases

| Phase | JSON Key | When Evaluated |
|-------|----------|---------------|
| HTTP Request | `on_http_request` | When an HTTP request arrives at the endpoint |
| HTTP Response | `on_http_response` | When an HTTP response is returned to the client |
| TCP Connect | `on_tcp_connect` | When a TCP connection is established |

## Key Operations

| Method | Description |
|--------|-------------|
| `NewTrafficPolicy()` | Creates a policy with empty rule arrays |
| `NewTrafficPolicyFromJSON(data)` | Parses JSON, validates top-level keys, returns error for unknown keys |
| `Merge(other)` | Appends all rules from another policy into this one |
| `IsEmpty()` | True if all phases have zero rules |
| `ContainsAction(actionType)` | Checks if any rule in any phase contains the given action type |
| `DeepCopy()` | Creates a deep copy via JSON round-trip |

## Builder Functions

The package provides constructors for common action types:

| Builder | Action Type | Description |
|---------|-------------|-------------|
| `NewAddHeadersAction(headers)` | `add-headers` | Add HTTP headers |
| `NewRemoveHeadersAction(headers)` | `remove-headers` | Remove HTTP headers |
| `NewCompressResponseAction(algorithms)` | `compress-response` | Response compression |
| `NewCircuitBreakerAction(...)` | `circuit-breaker` | Circuit breaker with thresholds |
| `NewCustomResponseAction(status, content, headers)` | `custom-response` | Hard-coded response |
| `NewOAuthAction(config)` | `oauth` | OAuth authentication |
| `NewOIDCAction(config)` | `openid-connect` | OIDC authentication |
| `NewRestricIPsActionFromIPPolicies(policies)` | `restrict-ips` | IP-based access control |
| `NewTerminateTLSAction(config)` | `terminate-tls` | TLS termination |
| `NewWebhookVerificationAction(provider, secret)` | `verify-webhook` | Webhook verification |
| `NewForwardInternalAction(url)` | `forward-internal` | Forward to internal endpoint |

## Generated Traffic Policies

The Manager Driver generates traffic policies during IR-to-endpoint translation:

- **Routing rules**: When a hostname has multiple upstreams, the Driver generates `on_http_request` rules with CEL expressions matching path/header/query criteria and `forward-internal` actions routing to the appropriate internal AgentEndpoints.
- **TLS termination**: When TLS configuration is present on an IRVirtualHost, a `terminate-tls` action is generated.
- **Header manipulation**: Per-route traffic policies from Ingress annotations or Gateway API filters are merged into the generated policy.
